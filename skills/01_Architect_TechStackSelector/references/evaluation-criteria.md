# 技术选型评估矩阵 (Evaluation Matrix)

| 维度 | 评估标准 | 考量点 |
| :--- | :--- | :--- |
| **SEO** | 搜索引擎可见度 | 是否支持 SSR/SSG？TBT/LCP 表现如何？ |
| **TTM** | 交付时间 (Time to Market) | 社区是否有成熟模版？开发工具链是否高效？ |
| **Scalability** | 水平扩展能力 | 是否容易拆分微服务？数据库连接池处理如何？ |
| **Real-time** | 实时交互能力 | 是否原生支持 WebSocket/SSE？ |
| **Security** | 安全性 | 认证鉴权（Auth）是否有现成方案？漏洞修复频率？ |
| **Cost** | 运营成本 | 托管费用、维护成本、学习曲线。 |
| **Talent Pool** | 人才储备 | 市场上该技术栈的开发者多吗？招聘难度如何？ |

## 决策逻辑 (Decision Logic)

1. **如果业务需要极佳的 SEO 且有大量静态内容**：
   - 优先选择 `Astro`。
2. **如果业务是复杂的 SaaS 且需要前后端一体化开发体验**：
   - 优先选择 `Next.js`。
3. **如果团队主要是 Python 开发者且需要高性能 API**：
   - 优先选择 `FastAPI`。
4. **如果是一个简单的后台管理系统且追求开发速度**：
   - 优先选择 `React + Refine` 或 `Vue + Element`。
5. **如果需要快速上线且不想管服务器**：
   - 优先选择 `Supabase` + `Vercel`。
